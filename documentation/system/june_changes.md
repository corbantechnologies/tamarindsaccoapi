# System Changes & Additions (June Updates)

This document tracks the functional improvements and architectural additions made to the SACCO API backend. These changes serve as a baseline for deploying and configuring other SACCO instances.

## 1. Comprehensive Audit Logging System
To ensure strict financial compliance and tracking, a robust, fully automated audit logging system was implemented across both the backend and frontend.

### Backend Infrastructure
- **Model Architecture (`auditlogs/models.py`)**: 
  - Created the `AuditLog` model inheriting from standard project abstracts (`UniversalIdModel`, `TimeStampedModel`, `ReferenceModel`).
  - Implemented PostgreSQL native `JSONField` types (`request_payload` and `response_payload`) to capture complete system state changes.
- **Asynchronous Middleware (`auditlogs/middleware.py`)**:
  - Built `AuditLogMiddleware` that automatically intercepts all state-changing HTTP requests (`POST`, `PUT`, `PATCH`, `DELETE`).
  - Parses and securely stores the exact incoming JSON requests and the exact outgoing JSON responses.
  - Automatically filters out sensitive keys (e.g., `password`, `pin`, `token`, `access`) before persisting payloads.
  - Offloads database writes to a background `threading.Thread` to guarantee zero impact on API latency.
- **Manual Logging Utility (`auditlogs/utils.py`)**:
  - Added a `log_action` utility function to allow developers to trigger explicit, highly descriptive logs from deep within complex business logic (e.g., inside loan approval services).

### Frontend Integration & Security
- **Superuser Namespace**: 
  - Bootstrapped a dedicated `/superuser/` routing path in Next.js to completely isolate system monitoring from day-to-day `/sacco-admin/` operations.
- **JSON Payload Viewers**:
  - Added a "View Details" modal to both the Superuser and Sacco Admin log tables.
  - Auditors can now click any log entry to instantly view a formatted, syntax-highlighted representation of the exact `request_payload` sent by a user and the `response_payload` generated by the system.

## 2. API-Driven Superuser Creation
To eliminate reliance on the `python manage.py createsuperuser` command line utility, a secure endpoint was introduced to handle superuser provisioning.

### Implementation Details
- **Endpoint**: `POST /api/v1/accounts/superusers/create/`
- **Security Logic (`accounts/permissions.py`)**:
  - Created a custom `CanCreateSuperuser` permission class to solve the "chicken and egg" problem.
  - **Bootstrap Mode**: If exactly `0` superusers exist in the database, the endpoint temporarily allows unauthenticated requests, permitting the very first "root" superuser to be created programmatically.
  - **Strict Lockdown**: The moment the first superuser is created, the endpoint instantly locks down and strictly requires the requestor to provide a valid token from an existing superuser account.
- **Serializer (`accounts/serializers.py`)**:
  - Implemented `SuperuserCreationSerializer` to securely force `is_superuser=True` and `is_staff=True` internally, ignoring any permission-escalation flags attempting to be passed from the client payload.
